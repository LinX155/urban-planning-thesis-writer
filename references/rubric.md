# Skill 自检 Rubric

逐项检查，任一阻断项未满足不得交付。

## 阻断项

- 是否明确写入：UPTW 在 Codex 中只暴露 3 个显式入口 `uptw-init`、`uptw-plan`、`uptw-write`，不依赖单体父 skill 路由。
- 是否明确写入：默认不自动启用，只在 `/UPTW-init`、`/UPTW-plan`、`/UPTW-write` 或对应显式 skill 调用时使用。
- 是否明确限制适用场景：用户已完成实验、分析、数据处理和主要结论，仅组织成长篇中文城市规划硕士论文 DOCX。
- 是否禁止：选题建议、实验设计、数据分析方法创造、结果编造、替用户虚构结论。
- 是否把质量审查、图表/公式/正文衔接、反 AI 模板腔、章节一致性、术语一致性、询问用户、备份机制作为内部机制，而不是额外用户可见 skill。
- 是否包含 Windows 可执行 init 流程和脚本依赖检测或安装。
- 是否支持 DOCX 快照、备份、差异摘要、用户修改保护和长期状态恢复。
- 是否基于全部 PDF 调研记录形成专业共性总结，并区分共性、少数亮点、不可泛化风格和不宜模仿写法。
- 是否避免复制论文原文、固定句式库或可能构成抄袭的模板。
- 是否不是普通论文写作指南，而是可复用 harness：包含状态目录、脚本、references、工作流和验证机制。
- 是否提供真实可安装产物：至少覆盖 Codex 与 Claude Code，能通过 `npm install -g` 自动安装 3 个全局 skill 入口，不需要额外 CLI bootstrap。

## 质量项

- `plan` 阶段是否要求读取用户现有材料、锁定事实边界、章节策略、术语、图表公式清单和开放问题。
- `write` 阶段是否明确增量写作、指定范围修改、保护已审定内容、更新状态和记忆。
- 是否要求图前说明用途、图后解释证据，公式前引入、公式后解释变量和用途。
- 是否要求学术表达避免空泛价值判断、过度分点、口语化、公众号化、重复总结和结论先行。
- 是否包含恢复机制：从状态目录和快照继续下一轮。
- README 是否面向用户说明安装、初始化、plan/write 使用和注意事项。
