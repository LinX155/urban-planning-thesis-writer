# Skill 自检 Rubric

逐项检查，任一阻断项未满足不得交付。

## 阻断项

- 是否明确写入：UPTW 在 Codex 中只暴露 2 个显式入口 `uptw-plan`、`uptw-write`，不依赖单体父 skill 路由。
- 是否明确写入：默认不自动启用，只在 `/UPTW-plan`、`/UPTW-write` 或对应显式 skill 调用时使用。
- 是否明确限制适用场景：用户已完成实验、分析、数据处理和主要结论，仅组织成长篇中文城市规划硕士论文 markdown。
- 是否禁止：选题建议、实验设计、数据分析方法创造、结果编造、替用户虚构结论。
- 是否把质量审查、图表/公式/正文衔接、反 AI 模板腔、章节一致性、术语一致性、询问用户、备份机制作为内部机制，而不是额外用户可见 skill。
- 是否包含 Windows 可执行 bootstrap 流程，并由首次 `/UPTW-plan` 自动触发脚本依赖检测或安装。
- 是否明确把首次 `/UPTW-plan` 拆成 `bootstrap / inventory / outline / briefs` 四个内部 phase，并要求 phase 之间落盘与恢复。
- 是否支持 markdown 快照、备份、差异摘要、用户修改保护和长期状态恢复。
- 是否提供基于”最小改动面”原则的 markdown 改写能力，而不是把工具选择固化成冗长的 case 列表。
- 是否要求首次 plan 先形成 `material_inventory.json`，而不是仅凭目录浏览直接进入章节设计。
- 是否基于全部 PDF 调研记录形成专业共性总结，并区分共性、少数亮点、不可泛化风格和不宜模仿写法。
- 是否避免复制论文原文、固定句式库或可能构成抄袭的模板。
- 是否不是普通论文写作指南，而是可复用 harness：包含状态目录、脚本、references、工作流和验证机制。
- 是否提供真实可安装产物：至少覆盖 Codex 与 Claude Code，能自动安装 2 个全局入口且首次 `/UPTW-plan` 不需要额外 CLI bootstrap。

## 质量项

- `plan` 阶段是否要求读取用户现有材料、锁定事实边界、章节策略、术语、图表公式清单和开放问题。
- `plan` 阶段是否要求对大范围请求进行 batch 化 brief 落盘，并在未完成时留下 resume 点而不是草率收束。
- `write` 阶段是否明确增量写作、指定范围修改、保护已审定内容、更新状态和记忆。
- 是否要求图前说明用途、图后解释证据，公式前引入、公式后解释变量和用途。
- 是否要求涉及公式时先用自然语言叙述推导逻辑，变量符号按需使用，简单量优先用中文描述。
- 是否要求文内公式首次出现的变量以括号解释，换行公式另起行逐一说明变量含义。
- 是否要求出现阈值时附带含义解释和取值理由。
- 是否要求检查中文表达的歧义性，确保每种表述只有一种理解方式。
- 是否要求学术表达避免空泛价值判断、过度分点、口语化、公众号化、重复总结和结论先行。
- 是否包含恢复机制：从状态目录和快照继续下一轮。
- README 是否面向用户说明安装、首次 plan 自动初始化、plan/write 使用和注意事项。
